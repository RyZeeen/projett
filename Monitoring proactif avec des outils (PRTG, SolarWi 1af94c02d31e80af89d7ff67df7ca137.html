🛡️ 2️⃣ IDS/IPS : Intrusion Detection & Prevention

a. 🔍 Objectifs IDS/IPS

  • Détection en temps réel des attaques réseau/applicatives
  • Prévention des intrusions en bloquant les flux malveillants
  • Visibilité sur :
    • Attaques réseau (scans, DoS)
    • Intrusions applicatives (SQLi, XSS)
    • Anomalies internes (latence, volumétrie anormale)

b. 🛠️ Déploiement

ZoneSolution recommandéeModePosition
SD-WANIDS/IPS intégré (ex. Fortinet, Palo Alto)Inline (prévention)Entre WAN ↔ LAN
Paris/Genève LANSuricata / Snort (open source)Passive (détection)SPAN/Mirror port switch
Cloud (Azure/AWS)- Azure: Microsoft Defender for Cloud- AWS: GuardDuty + IDS partenaire (ex : Trend Micro, Suricata)Inline (Azure/AWS Gateway)Cloud front-end + réseau interne

c. 🔗 Intégration & supervision

  • Centralisation des logs : SIEM (Splunk, ELK)
  • Alerting : Zabbix / Grafana + email & SMS
  • Corrélation : liaison avec EDR cloud & logs applicatifs

d. 🔄 Maintenance & test

  • Règles IPS : mise à jour auto (Suricata, fournisseurs)
  • Tests d’efficacité : Metasploit, Kali Linux (trimestriel)
  • Journalisation : incidents bloqués, faux positifs, ajustements des règles

🏗️ 3️⃣ Segmentation Zero Trust (ZTNA + LAN)

a. 🔎 Objectifs ZTNA

  • Interdire toute connexion par défaut ("deny all")
  • Autoriser uniquement les flux strictement nécessaires
  • Appliquer une politique : authentification + évaluation du contexte + politique réseau

b. 🖧 Segmentation réseau physique (Paris & Genève)

ZoneVLAN/ZoneExemple
VLAN 10AdministratifServeurs internes
VLAN 20Utilisateurs métiersPostes utilisateurs
VLAN 30InvitésAccès Internet uniquement
VLAN 40IoT / imprimantesPas d'accès serveur
VLAN 50Sécurité (caméras, alarme)Accès isolé
DMZServices exposés (web, email)Isolé du LAN
  • Pare-feu inter-VLAN strict (ex : deny all par défaut + règles fines)
  • Micro-segmentation : sur les serveurs sensibles (ex : pare-feu local Azure NSG / AWS SG)

c. 🌐 Segmentation cloud (Azure & AWS)

  • Azure NSG (Network Security Groups)
  • AWS Security Groups + ACL réseau
  • Principes :
    • Pas d’ouverture globale (0.0.0.0/0) sauf nécessité contrôlée
    • Segmentation par rôle applicatif (front-end ↔ back-end ↔ DB)
    • Journalisation des accès (logs NSG / VPC Flow Logs)

d. 🔐 ZTNA pour utilisateurs

  • Authentification MFA (Okta, Azure AD)
  • Contexte d’accès :
    • Postes approuvés uniquement (compliance Intune)
    • Vérification de la géolocalisation / adresse IP
  • Contrôle de session : autorisation dynamique (CSP, inspection TLS)

👥 4️⃣ Gestion des accès par rôle (RBAC)

a. 🏛️ Objectifs

  • Principe du moindre privilège
  • Attribution des droits en fonction des rôles métier
  • Évolutivité : adaptation des rôles lors des changements

b. 📋 Définition des rôles

RôleExemples de droits
Admin ITPlein accès infra + cloud (root, admin Azure/AWS)
Support ITAccès limité aux outils de support (monitoring, tickets)
Utilisateurs métierAccès aux applications métier seulement
Invité / TiersAccès restreint à une ressource spécifique (ex : audit externe)

c. 🔑 Mise en œuvre

ZoneTechnologieExemple
AD localGroupes ADUtilisateurs ↔ Serveurs ↔ Imprimantes
Cloud AzureRBAC AzureOwner, Contributor, Reader (par ressource)
AWSIAMRoles/Policies : ex "EC2ReadOnly", "S3FullAccess"
ZTNAPoliciesPostes approuvés ↔ Ressources précises

d. 🚦 Contrôles & supervision

  • Revue trimestrielle des droits
  • Audit automatique : rapports Azure IAM / AWS IAM
  • Journalisation des accès sensibles (CloudTrail, Azure Logs)

6️⃣ Mise en production

🚨 6️⃣ Alerting automatique

a. 🔔 Seuils d’alerte

MétriqueSeuil d’alerte
CPU serveur>85% sur 5 minutes
Espace disque<15% libre
Perte WAN>5% perte sur 2 minutes
Latence WAN>100ms
IDS/IPSDétection critique immédiate
Cloud (VM down)Immédiate

b. 📤 Canaux d’alerte

  • Email : équipe IT
  • SMS : astreinte H24
  • Teams/Slack : webhook direct sur canal d’incident
  • Ticket auto : via ServiceNow ou autre ITSM

c. 🛑 Gestion des fausses alertes

  • Seuils ajustés après 1 mois d’observation
  • Escalade progressive :
    • Alerte 1 : mail uniquement
    • Alerte 2 (si non résolue 10 min) : SMS + ticket
  • Mode maintenance : éviter les faux positifs pendant les MAJ

🚀 Exemple de workflow : détection panne WAN Paris

1️⃣ SD-WAN détecte >5% perte → PRTG déclenche alerte

2️⃣ Email + Teams envoyés au canal incident

3️⃣ Si non résolu sous 10 min → SMS + ticket ITSM

4️⃣ Technicien bascule temporairement le WAN sur le backup

5️⃣ Rapport automatique post-incident (root cause + mesures)
