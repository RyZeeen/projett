Documentation des procédures en cas d'incident réseau ou d'intrusion (PRA/PCA)

PRA/PCA structuré, avec un focus sur la traçabilité, la réactivité et la conformité.

🎯 Objectif

Mettre en place une documentation claire, accessible et maintenue à jour pour répondre efficacement aux incidents réseau et intrusions, avec intégration dans le PRA (Plan de Reprise d’Activité) et le PCA (Plan de Continuité d’Activité).

🧱 Contexte de l’infrastructure

  • Sites de Paris et Genève reliés par SD-WAN
  • Utilisation de ZTNA
  • Environnement hybride Azure / AWS / On-Prem
  • Données critiques et utilisateurs répartis
  • Besoin de garantir disponibilité, intégrité et confidentialité

🧠 1. Principes de documentation

PrincipeObjectif
ClartéCompréhension immédiate en cas de stress ou d’urgence
CentralisationRegroupement dans un espace sécurisé partagé
Mise à jour régulièreConformité avec l’infra réelle et les outils
Accessibilité contrôléeAccès restreint à l'équipe IT/Sécurité uniquement

📚 2. Types de documents à créer

a. 📄 Procédures d’incident réseau

Exemples de scénarios :

  • Perte d’un lien SD-WAN
  • Saturation de bande passante
  • Déconnexion VPN / ZTNA
  • Panne de switch/firewall
  • Boucle réseau

Structure recommandée :

markdown
CopierModifier
Titre : Incident [Type]
But : Identifier, isoler, corriger
Étapes :
  1. Détection (supervision, logs, ticket)
  2. Diagnostic (ping, traceroute, SNMP, dashboard)
  3. Contournement temporaire (bascule WAN, reroutage)
  4. Résolution
  5. Communication interne
  6. Enregistrement de l’incident
Responsable : Admin réseau / Infra
Dernière mise à jour : JJ/MM/AAAA

b. 🔐 Procédures en cas d’intrusion / cyberattaque

Types :

  • Ransomware
  • Tentative de phishing réussie
  • Compromission de compte
  • Malware sur poste
  • Mouvements latéraux internes

Structure recommandée :

markdown
CopierModifier
Titre : Réponse à une tentative d’intrusion [Scénario]
Objectif : Contenir et éradiquer la menace
Étapes :
  1. Détection (EDR, SIEM, alerte ZTNA)
  2. Isolement de l’équipement (NAC, VLAN, coupure WiFi)
  3. Analyse des logs
  4. Remédiation (scan, reimaging, patch)
  5. Notification interne (RSSI, DSI, juridique)
  6. Documentation de l'incident
  7. Déclaration CNIL / ANSSI (si données sensibles)
Responsable : Équipe Sécurité
Dernière mise à jour : JJ/MM/AAAA

🔐 3. Intégration PRA/PCA

a. Lien avec PRA

  • Chaque procédure d’incident → rattachement au scénario PRA concerné
    • Ex : "Perte de lien SD-WAN Paris ↔ Genève" → Scénario PRA #2
  • Checklist PRA incluse à la fin de chaque procédure
  • Intégration dans le runbook PRA

b. Lien avec PCA

  • Pour incidents n'interrompant pas les services, mais impactant leur qualité
    • Ex : perte de redondance, hausse de latence
  • Triggers pour activer PCA partiel

🗂️ 4. Stockage et format

FormatUsage
.docx ou .mdDocumentation lisible par l’équipe IT
Wiki interne (Confluence, Wiki.js…)Consultation rapide et versionnée
Dossier partagé sécurisé (Nextcloud, OneDrive Entreprise)Accès rapide, sauvegardé

Bonnes pratiques :

  • Versionner chaque procédure
  • Horodatage de la dernière révision
  • Ajout d’un auteur/responsable référent
  • Lier avec des tickets JIRA / ServiceNow / GLPI

🧪 5. Maintenance & test

  • Audit documentaire chaque semestre
  • Simulation d’incidents tous les 6 mois (phishing, panne SD-WAN simulée…)
  • Post-mortem obligatoire avec mise à jour de la procédure
  • Checklist d’amélioration continue

🔧 6. Modèle de procédure à répliquer

markdown
CopierModifier
# 🚨 Procédure : Intrusion détectée sur un poste utilisateur

**But** : Isoler rapidement le poste, limiter les dégâts, identifier l'origine.

**Déclencheur** : Alerte CrowdStrike / SentinelOne sur comportement anormal

### Étapes
1. Alerte reçue sur [outil de SIEM] → vérification de l’alerte
2. Isolement du poste via [NAC / ZTNA / VLAN]
3. Analyse des journaux (dernier login, connexions sortantes, mouvements latéraux)
4. Restauration image propre si compromise
5. Notification du RSSI / DSI
6. Rapport incident envoyé dans les 24h

**RTO associé** : 1h
**Lien PRA** : Scénario #4
**Dernière révision** : 05/04/2025
**Responsable** : Référent Sécurité / IT Ops
